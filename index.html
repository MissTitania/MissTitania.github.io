I'm Titania, a Berkeley programmer who studied computer science at Caltech. My coding experience is primarily with Java and Python. Various interests include machine learning/AI, web development, and information theory. You can browse this site to check out a couple of my coding projects!

Some older undertakings don't appear here. In high school I worked on a bridge bidding AI. That's always been a challenging problem for computers. Traditional approaches aren't well-equipped to handle several oddities: imperfect information, cooperative agents, and a titanic game-space. To this day, programs are inadequate for moderate-level competition. My algorithm broaches these with Monte Carlo simulation. The main hurdle is to foresee potential outcomes from both teammates' limited perspectives. I achieved this by drawing new hand samples for auction look-aheads at every node of the search tree. Then expectation evaluation can occur at each depth in a way that remains independent from that sub-tree's progenitor sample. This corrects for agents assuming others will interpret claims with the same confidence level that they do. Terminal outcomes were rated with a double-dummy solver that I implemented. The result was powerful enough to consistently defeat random Bridge Base Online opponents!

I also made attempts in the Netflix Contest and followed related published papers. Methods were all rooted in stochastic gradient descent among extracted feature vectors. I employed k-fold cross-valuation, baseline bias corrections, and regression synthesis with other models. While I did significantly out-perform Netflix's software, I fell just short of requisite performance gains to win.

I have a strong math background. In fact, previously in a public high school I taught it! I found it personally rewarding to restructure ideas to make new information intuitive. The whole experience was definitely a great developer for my communication skills.

This is an in-browser AI implementation for the classic game Othello. If you haven't played before, I recommend you start here with the rules.

This AI finds the best obtainable position by minimax search and alpha-beta pruning. Notably absent is a transposition table; the collision rate in Othello is too low to be worth the overhead. Mid-game states are evaluated using a function pre-computed through neural networks. Through self-play, these train coefficients for perfect-play results of all rectangular board sub-games.

The neural nets are coded and trained separately in Java. This allows a powerful numerical computation technique Javascript prohibits: bitboard representations. The technique stores boards as 2 64-bit integers (1 for each player) marking all piece positions with a 1 bit. With some cleverness, this allows calculation of all relevant features in extremely few clock cycles. Bitwise operations and precomputed masks yield board subsets, legal moves, and flipped pieces.

Pokemon Showdown is a community for competitively-minded Pokemon battling as a strategy game. The Azure server had readability, aesthetics, and ease of use problems and desperately required an interface remodel. To that end they hired me to remake their CSS, but with a catch! Esoteric community license agreement rules only allowed them to modify server-side code. They must use the verbatim same client as the rest of the community, differing only by a style sheet. So my task was to modernize the interface without access to the underlying Javascript or DOM. The result contained a great deal of animation for both a sleek-looking product and intuitive control element identification. This means that for an authentic experience I would encourage you to try injecting the style sheets yourself and see it in action! Still pictures can only tell so much.

For starters, PS is a battle-based entity, so let's cut right to the chase with the battle interface:

Moving the team and name display sidebars up top creates a lot more space to show off the actual battle! The graphical space in the center is where all the animations play out, so space there is at a premium. But the biggest gain of all isn't visible in a still picture. See that little notch at the bottom of the re-designed battlefield? As moves are made, now they're called out by player sprites in speech bubbles right there. Beforehand all these actions were merely dumped elsewhere in a chat log. Surely one of the best ways to ruin a child's fun is forcing them to scroll through textual data dumps, so the change was very much overdo. Next up is private conversations:

Even a few short conversations quickly overwhelm the space they're allocated. Only very intermittently does that menu on the left do anything; by and large it just eats up direct message space. Given that, it makes sense to have it be retractable. Then we can have luxurious wide conversations and battle challenges. And the new direct message boxes offer smart benefits: subtley brightening to display notifications, more intelligent vertical sizing, and minimization. It's all aimed at allowing as many of these simultaneous conversations as possible!

Here's what that main menu looks like when slid out. It now centers properly, and comes out further than before for larger print. And perhaps the most overdo: the main menu is finally condensed into just one menu. Instead of three. Why those similarly-purposed features were broken up before is hard to figure. And because it's a server for battling, now the battle options are adorned with slick animations and flashy mouseover icons. After all, user activity and hype ought to align with intended purpose!

Here's a public chatroom. Before, conversations were punctuated with various polls, server declarations, tournament updates, and more. These can be made much less obtrusive with some reformatting and better opacity use. As a bonus, now we can see the server mascot without so much junk on top! Overall, darkening the color scheme and reserving bright colors for important content helps highlight important information.

I do encourage you to check it out by injecting the CSS yourself! Seeing and using all the components makes for an experience still images cannot. If you do, don't forget it all happened without changing any code! Everything's achieved solely through style sheets.
